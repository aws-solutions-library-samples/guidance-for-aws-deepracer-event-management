# Race Results PDF — Design Spec

## Goal

Three server-side generated PDFs for a DREM event:

1. **Organiser summary** — full event leaderboard for the event organiser/sponsor
2. **Racer certificate** — per-racer "certificate of achievement" with their placement and stats
3. **Podium** — top 3 racers on a podium graphic with a footer table of positions 4–10

PDFs are generated on demand via AppSync, stored in S3, delivered to the user via a pre-signed URL.

## Architecture

### New CDK construct: `RaceResultsPdf`

Lives at `lib/constructs/race-results-pdf.ts`. Creates:

- **PDF S3 bucket** (`drem-backend-{label}-pdfs-*`)
  - Encryption: AWS-managed
  - Lifecycle rule: expire objects after 1 day. Pre-signed URLs are valid for 1 hour, so anything older than that is unreachable anyway — the extra day is a safety margin for edge cases (clock drift, async debugging). Users who need a PDF after expiry re-trigger generation.
  - No public access; delivery via pre-signed URL only
- **PDF Lambda** (`lib/lambdas/pdf_api/`) — Python 3.12 ARM64
  - Uses WeasyPrint + Jinja2 to render HTML templates to PDF
  - Reads race + event data from the RaceManager and EventsManager DynamoDB tables (same pattern as the stats EVB Lambda — tables exposed as public properties on their constructs, passed to the PDF construct)
  - Writes PDF to S3, returns pre-signed URL
  - Timeout: 2 minutes (enough for bulk certificate zip generation)
  - Memory: 512 MB (WeasyPrint benefits from headroom)
- **WeasyPrint Lambda layer**
  - Packaged separately because WeasyPrint + cairo + pango + pangoft2 is large
  - Reusable if other features need PDF generation later
- **AppSync schema additions** — a single mutation `generateRaceResultsPdf` with a `PdfType` enum discriminator

### Lambda layer build

WeasyPrint's native dependencies (cairo, pango, pangoft2, glib) don't ship with Lambda's default runtime. Build the layer via the project's existing Docker-based bundling pipeline (same pattern `ClamscanServerless` uses for the ClamAV binary). Reference: the `awslambda-hacks` community has working Dockerfiles for WeasyPrint on Python 3.12 ARM64 that install the native libs into `/opt/lib` and set `LD_LIBRARY_PATH`.

### Auth model

| PDF | Auth | Notes |
|-----|------|-------|
| Organiser summary | Cognito — admin/operator/commentator | Same as race admin |
| Bulk certificates (all racers in an event, zip) | Cognito — admin/operator | Could be slow; run async if needed |
| Per-racer certificate | Cognito — admin/operator OR the racer themselves | Self-service: racer can only generate their own |

The per-racer-self-service check happens in the Lambda: compare the authenticated user's `sub` against the requested `userId`. Admin/operator/commentator bypass this check.

(Public, no-auth shareable certificate URLs via API key are **out of scope** for v1 — reconsider when the racer stats page lands, since the two share the same auth pattern.)

## AppSync schema

```graphql
enum PdfType {
  ORGANISER_SUMMARY   # whole event leaderboard, multi-page
  PODIUM              # single page, top 3 + footer table
  RACER_CERTIFICATE   # single page per racer, letter/A4 landscape
  RACER_CERTIFICATES_BULK  # zip of all racers' certificates for an event
}

type PdfGenerationResult {
  downloadUrl: String!   # pre-signed S3 URL, 1-hour expiry
  filename: String!      # suggested filename for download
  expiresAt: AWSDateTime!
}

type Mutation {
  generateRaceResultsPdf(
    eventId: ID!
    type: PdfType!
    userId: ID              # required for RACER_CERTIFICATE
    trackId: ID             # optional — restrict to one track for multi-track events
  ): PdfGenerationResult @aws_cognito_user_pools(cognito_groups: ["admin", "operator", "commentator", "racer"])
}
```

Separate mutations per PDF type would bloat the schema. One mutation with a type discriminator is simpler; the Lambda routes internally.

## PDF content

### 1. Organiser Summary (multi-page A4 portrait)

- **Header**: DeepRacer logo (top-left), event name (large), event date + country flag + type + sponsor (subheader)
- **KPI row**: 4 boxes — total racers, total races, valid laps, overall fastest lap
- **Leaderboard table**: rank, racer, country flag, fastest lap, avg lap, valid/total laps, most consecutive laps, raced-by-proxy indicator, sorted by ranking method (fastest lap or best-average-lap)
- **Per-track sections** (only if event has >1 track)
- **Footer**: "Generated by DREM on {date}" + small DeepRacer logo

### 2. Racer Certificate (A4 landscape, single page)

- Decorative border (CSS — no image asset needed)
- DeepRacer logo (top centre)
- "Certificate of Achievement" heading
- Racer name (large, centred)
- Body: "Participated in **{eventName}** on **{eventDate}** at **{location}**"
- Result row: rank badge (e.g. "3rd place"), fastest lap, most consecutive laps
- Signature line / space for sponsor logo (left side, optional)
- Footer: DREM small mark + generation date

### 3. Podium (A4 portrait, single page)

- Header: DeepRacer logo + event name/date
- **Podium graphic**: 3 steps rendered in CSS (gold/silver/bronze), names and country flags inside each step. Gold step is tallest (centre), silver left, bronze right — standard podium layout
- Below: table for positions 4–10 with name, country, fastest lap
- Footer: "Full results available on DREM" + generation date

## Template structure

Jinja2 HTML templates under `lib/lambdas/pdf_api/templates/`:

```
templates/
  base.html                 # shared layout, @page CSS, footer
  _components.html          # Jinja macros: kpi_box, racer_row, podium_step
  organiser_summary.html
  racer_certificate.html
  podium.html
  static/
    deepracer-logo.png      # default logo (bundled)
    certificate-border.svg  # decorative certificate border
```

All templates extend `base.html` which defines:
- `@page` rules (A4 portrait/landscape, margins, footer)
- CSS variables for colours (defaults set here; #52 white-labelling overrides)
- Imported fonts (Amazon Ember if licensing allows, else system sans-serif)

Brand-customisable values (logo URL, primary colour, accent colour, visible field set) are passed from the Lambda as context. For v1 these come from hardcoded defaults in the Lambda; #52 will add a config lookup.

## Data flow

1. User clicks "Download PDF" in the UI
2. Frontend calls `generateRaceResultsPdf` mutation with eventId + type
3. Lambda:
   - Reads event record from Events table
   - Reads races from Race table (filtered by eventId + optional trackId)
   - For per-racer: looks up racer username + country from Cognito
   - Computes the race summary (ranking, fastest lap, avg lap, etc.) — reuses the logic from `race_api`'s `__calculate_race_summary`
   - Renders the Jinja2 template to HTML
   - WeasyPrint renders HTML → PDF bytes
   - Uploads to S3 with key `{eventId}/{type}/{yyyy-mm-dd-hhmmss}.pdf`
   - Generates pre-signed URL (1 hour expiry)
   - Returns `PdfGenerationResult`
4. Frontend opens the download URL in a new tab (or triggers a download)

### Bulk certificates

`RACER_CERTIFICATES_BULK` path:
1. Read all races for the event, group by userId
2. Render one certificate PDF per racer
3. Zip into a single archive
4. Upload zip to S3
5. Return pre-signed URL to the zip

For large events (100+ racers) this can take 30–90s. If it exceeds the 2-minute Lambda timeout, move to an async pattern via EventBridge + notification — but YAGNI until we see it happen.

## Compute reuse

The ranking + summary logic duplicates `lib/lambdas/race_api/__calculate_race_summary`. Two options:

- **Extract to a shared Lambda helper layer** (new or added to `helperFunctions`) — cleanest, but touches multiple constructs
- **Port and re-implement** in the PDF Lambda — easier for v1, risk of drift

For v1, re-implement (same approach we took with the stats compute module). The shared-layer refactor is a later cleanup — noted as a follow-up.

## Frontend

- **Race admin page** (`admin/race-admin/raceAdmin.tsx`) — add a "Download PDF" split button per event with three options: "Organiser summary", "Bulk certificates (zip)", "Podium"
- **Racer profile / timekeeper flows** — add "Download certificate" button where a racer's results are shown (admin side for now; racer-self-service added when #33 public racer profile lands)

Frontend uses the existing `graphqlMutate` helper with the new mutation, then calls `window.open(downloadUrl)` (or triggers a download) on success.

## Failure modes

- **No races for event** — return a validation error ("Cannot generate PDF for an event with no races")
- **Unknown eventId / userId** — return a validation error
- **Deleted Cognito user with race history** — the userId (Cognito sub) still exists in race records but Cognito `adminGetUser` returns not-found. Fall back to the `username` cached on the leaderboard entry (or `sub[:8]` if even that's missing). Never fail PDF generation because a user was deleted.
- **WeasyPrint render fails** — log to CloudWatch, return a generic error ("PDF generation failed, please try again")
- **S3 upload fails** — CloudWatch log + retry once; second failure returns error
- **Pre-signed URL expiry** — 1 hour default; if the user waits longer they can re-trigger the mutation

## Testing

- **Unit tests**: template rendering — given a fixture event + races dict, render each template and assert known strings appear (racer names, event name, lap times). Fast, no WeasyPrint needed.
- **Integration test**: run once during development against a real event in the dev deployment; eyeball the PDF. Not automated.
- **CDK tests**: `RaceResultsPdf` construct synthesises cleanly, S3 bucket + Lambda + layer created.

## Out of scope (for v1)

- Logo upload, colour scheme customisation, field visibility (task #52 white-labelling)
- Public shareable certificate URLs (revisit with racer stats page #33)
- Async generation for very large bulk jobs (YAGNI until we see timeouts)
- Email delivery (user downloads the URL and emails themselves)
- I18n of PDF content (English only for v1; copy structure so translation keys can be added later)

## Known redesign risks

The user flagged that this spec will likely need revisiting when:

- **PR #171 (avatar + highlight colour) merges** — certificates could include the racer's avatar and use their highlight colour for accents
- **#29 (overlay redesign) lands** — the overlay team will settle on event branding patterns that we should match here
- **#52 (white-labelling) is built** — replaces hardcoded DeepRacer defaults with tenant-configurable logo + colours + fields

Plan for this: keep brand values (logo URL, colours, field visibility) as a single config object the templates read from. v1 hardcodes it; #52 swaps in a runtime lookup. Templates shouldn't need structural changes.
