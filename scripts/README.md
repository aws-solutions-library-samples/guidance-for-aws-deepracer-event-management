# DREM Data Export/Import Tools

Export and import DREM instance data — events, races, leaderboard entries, fleets, landing page configs, racer profiles (avatar + highlight colour), and Cognito users. Useful for migrating data between environments, seeding dev instances, or backing up production data.

## Prerequisites

```bash
make venv   # or: make local.config.python
source .venv/bin/activate
```

Run all commands from the **repo root** (where `build.config` and `cfn.outputs` live).

## Export

### From your own DREM instance (DynamoDB mode)

Requires AWS credentials with access to the deployed stack. Table names are auto-discovered from CloudFormation.

```bash
# Full export (events, races, leaderboard, fleets, landing pages, Cognito users)
python scripts/drem_export.py

# Custom output directory
python scripts/drem_export.py --output ./my-backup/

# Data only (skip Cognito users)
python scripts/drem_export.py --skip-users

# Export specific events only
python scripts/drem_export.py --events evt-abc123,evt-def456

# Target a different stack label (default reads from build.config)
python scripts/drem_export.py --stack dev
```

### From a remote DREM instance (API mode)

Only needs a valid admin JWT token and the AppSync endpoint URL. No AWS infrastructure access required.

```bash
python scripts/drem_export.py \
  --api \
  --endpoint https://xxx.appsync-api.eu-west-1.amazonaws.com/graphql \
  --token "eyJraWQ..."
```

To get the endpoint URL: open the DREM website in a browser, open DevTools, filter Network requests for "graphql", and copy the request URL.

To get the token: log in to DREM as an admin user, open DevTools, find the `Authorization` header on any GraphQL request, and copy the JWT value.

**Note:** Tokens expire after 1 hour. The API mode may not export leaderboard data if the remote instance has events with missing track configuration (a known server-side bug). Racer profiles in API mode are fetched per-username (no list query exists), so export time scales with the user count — DDB mode is faster for full backups.

### Export directory structure

```
drem-export-YYYY-MM-DD-HHMMSS/
  manifest.json          # export metadata, source stack, counts
  events.json            # event configurations
  races.json             # race records grouped by eventId
  leaderboard.json       # pre-computed leaderboard entries
  fleets.json            # fleet-to-car mappings
  landing_pages.json     # per-event landing page configs
  racer_profiles.json    # avatar config + highlight colour per username
  users.json             # Cognito user records with group memberships
```

## Import

Writes data directly to DynamoDB tables. Requires AWS credentials with write access to the target stack.

```bash
# Full import
python scripts/drem_import.py --input ./drem-export-xxx/

# Preview without writing anything
python scripts/drem_import.py --input ./drem-export-xxx/ --dry-run

# Skip Cognito user creation (use when importing into the same environment
# or when users already exist)
python scripts/drem_import.py --input ./drem-export-xxx/ --skip-users

# Skip leaderboard (it can be recomputed from race data)
python scripts/drem_import.py --input ./drem-export-xxx/ --skip-leaderboard

# Target a different stack
python scripts/drem_import.py --input ./drem-export-xxx/ --stack dev
```

### Cognito user handling

When importing users (`--skip-users` not set):

- Users are created with `admin_create_user` and a temporary random password
- Users land in **FORCE_CHANGE_PASSWORD** state — they must set a new password on first login
- Group memberships (admin, operator, commentator, etc.) are preserved
- The old Cognito `sub` (userId) is remapped to the new `sub` across all DynamoDB records
- The `old_sub -> new_sub` mapping is saved to `manifest.json` for auditability
- If a user already exists (username match), the existing `sub` is used for remapping

### What gets remapped

When user mapping is active, these fields are rewritten during import:

| Table | Fields remapped |
|-------|----------------|
| Races | `userId`, `sk` (contains `USER#{userId}`) |
| Leaderboard | `userId`, `sk` (contains `{trackId}#{userId}`) |
| Events | `createdBy` |
| Fleets | `createdBy` |
| RacerProfile | _none — keyed by `username`, which Cognito preserves on user recreation_ |

## Rebuild Statistics

Direct DynamoDB writes bypass EventBridge, so the statistics engine won't automatically recompute after an import. Run the rebuild script to trigger it:

```bash
python scripts/drem_rebuild_stats.py
```

This discovers the Stats EVB Lambda via CloudFormation, finds a real event in the database, and invokes the Lambda to trigger a full global stats rebuild. Takes a few seconds for typical deployments.

## Table Discovery

All three scripts auto-discover DynamoDB table names using:

1. `build.config` — for the stack label and region
2. `cfn.outputs` — for the Cognito user pool ID (generated by `make local.config`)
3. CloudFormation `list_stack_resources` — maps logical resource IDs to physical table names

Use `--stack <label>` to override the label from `build.config` (e.g., `--stack dev` to target `drem-backend-dev-infrastructure`).

## Data scope

| Source | Exported | Imported | Notes |
|--------|----------|----------|-------|
| Events | Yes | Yes | Core event config, tracks, race config |
| Races | Yes | Yes | Race records with laps and averageLaps |
| Leaderboard | Yes | Yes | Pre-computed; also recomputable from races |
| Fleets | Yes | Yes | Fleet-to-car mappings |
| Landing pages | Yes | Yes | Per-event landing page links |
| Cognito users | Yes | Yes | With group memberships; passwords not transferable |
| Statistics | No | No | Derived — use `drem_rebuild_stats.py` after import |
| Car status | No | No | Device-specific, not transferable |
| Models | No | No | S3 refs won't transfer between accounts |
| Car log assets | No | No | S3 refs won't transfer |
